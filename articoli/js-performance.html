Performance JavaScript: strategie per siti più veloci

JavaScript

Pubblicato il 28 giugno 2025 • Aggiornato il 13 agosto 2025 • ~15 min lettura

Il JavaScript è spesso il fattore più costoso per interattività e rendering. In questa guida completa trovi strategie pratiche per ridurre il costo del JS (peso, parsing, compilazione, esecuzione), migliorare i Core Web Vitals e mantenere alte le performance nel tempo.

Perché il JS impatta sui Core Web Vitals

Script grandi o eseguiti in momenti sbagliati bloccano il thread principale. Questo si riflette su:

  • LCP (Largest Contentful Paint): JS di blocco ritarda il rendering dell’elemento principale.
  • INP (Interaction to Next Paint): handler lenti o lunghi Long Tasks peggiorano la reattività.
  • TBT (Total Blocking Time): misura quante finestre > 50 ms impediscono input e paint.

Obiettivo: meno codice da scaricare, meno codice da eseguire, task più brevi e ben schedulati.

Misura: dove stai perdendo tempo

  • Lighthouse per una vista sintetica e raccomandazioni.
  • DevTools Performance: cerca Long Tasks > 50 ms, costi di Evaluate Script, Parse, Layout.
  • Monitora LCP, INP e TBT con web-vitals in produzione. 
// Esempio (web-vitals)
import { onLCP, onINP, onCLS } from 'web-vitals';
onLCP(console.log);
onINP(console.log);
onCLS(console.log);

Annota gli script più pesanti, i bundle più grandi e le interazioni con maggiore latenza (es. ricerca, menu, caroselli).

Riduci e spezza il codice

  • Rimuovi dipendenze non usate. Valuta alternative più leggere (dayjs > moment, Preact > React, librerie vanilla).
  • Tree-shaking: usa ES modules e imposta sideEffects correttamente nel package.json.
  • Code splitting: split per route, per feature e per vendor. Carica on‑demand con import().
  • Polyfill mirati: evita polyfill globali, usa polyfill.io o feature detection. 
// ESM + sideEffects
// package.json
{
  "type": "module",
  "sideEffects": ["*.css"]
}

Caricamento: defer, async, module/nomodule, preload/prefetch

  • <script defer> non blocca l'HTML, esegue in ordine; preferibile per script non critici.
  • <script async> scarica/esegue appena pronto; attenzione all'ordine.
  • type="module" è defer di default; abbinalo a nomodule per fallback legacy.
  • preload/prefetch: ottimizza la priorità delle risorse.
<link rel="preload" as="script" href="/app.js">
<script type="module" src="/app.js" defer></script>
<script nomodule src="/legacy.js" defer></script>

Lazy loading e caricamento condizionale

Carica componenti solo quando servono (on‑demand) o quando entrano nel viewport.

// IntersectionObserver + import dinamico
const el = document.querySelector('#widget-grafico');
const io = new IntersectionObserver(async (entries) => {
  if (entries.some(e => e.isIntersecting)) {
    const { initChart } = await import('./chart.js');
    initChart(el);
    io.disconnect();
  }
});
io.observe(el);

Per immagini/iframe usa loading="lazy" e decoding="async".

Eventi ad alta frequenza: debounce, throttle, passive

Riduci il lavoro eseguito durante scroll, resize, input.

// debounce
function debounce(fn, delay = 300) {
  let t; return (...args) => { clearTimeout(t); t = setTimeout(() => fn(...args), delay); };
}
// throttle
function throttle(fn, wait = 200) {
  let t = 0; return (...a) => { const n = Date.now(); if (n - t >= wait) { t = n; fn(...a); } };
}
// passive listeners
window.addEventListener('scroll', throttle(onScroll, 100), { passive: true });
// rAF per animazioni fluide
function onScroll() { requestAnimationFrame(updateParallax); }

Scheduling e lavori in background

  • requestIdleCallback: esegui attività non critiche quando il browser è idle.
  • scheduler.postTask: priorità fine‑grained dove supportato.
  • Web Workers: sposta CPU‑bound off‑thread (parsing, calcoli, trasformazioni). 
// requestIdleCallback
requestIdleCallback(() => preloadLessCriticalModules());

// Web Worker semplice
// main.js
const worker = new Worker('/worker.js', { type: 'module' });
worker.postMessage({ type: 'compute', payload: bigArray });
worker.onmessage = (e) => render(e.data);

// worker.js
self.onmessage = (e) => {
  if (e.data.type === 'compute') {
    // lavoro pesante qui
    self.postMessage(process(e.data.payload));
  }
};

Rendering e framework: SSR/CSR/ISR, hydration e islands

  • SSR: HTML pronto velocizza LCP; idrata solo dove serve.
  • Islands/Partial hydration: invia JS per componenti interattivi, non per tutta la pagina.
  • Streaming e resumability: riducono JS al client e tempi di attesa.

Evita idratazioni globali quando bastano micro‑interazioni locali.

Performance budget e monitoraggio continuo

    • RUM: invia i Vitals reali a un endpoint (es. /rum) e confronta per release.
    • CI: Lighthouse CI/WebPageTest con soglie minime e regressioni bloccanti. 
    // RUM beacon (esempio minimale)
import { onLCP, onINP, onCLS } from 'web-vitals';
function send(metric) {
  navigator.sendBeacon('/rum', JSON.stringify(metric));
}
onLCP(send); onINP(send); onCLS(send);

    # lighthouse-ci (lighthouserc.json)
{
  "ci": {
    "collect": { "numberOfRuns": 3, "staticDistDir": "dist" },
    "assert": {
      "assertions": {
        "performance": ["error", {"minScore": 0.9}],
        "total-blocking-time": ["error", {"maxNumericValue": 200}],
        "largest-contentful-paint": ["error", {"maxNumericValue": 2500}]
      }
    }
  }
}
  • Definisci budget: peso JS < 170 KB gz, TBT < 200 ms, INP < 200 ms.
  • Metti i budget in CI (Lighthouse CI, WebPageTest) e fallisci la build se superi le soglie.
  • Traccia i Vitals reali con RUM e confronta per release.

Esempi completi

// Split per route
async function bootstrapRoute(route) {
  if (route === 'dashboard') {
    const [{ initChart }, { initTable }] = await Promise.all([
      import('./features/chart.js'),
      import('./features/table.js')
    ]);
    initChart();
    initTable();
  }
}

HowTo: piano d’azione in 3 passi

  1. Misura con Lighthouse/DevTools e individua Long Tasks.
  2. Riduci e spezza (rimozioni, split, defer/async, import dinamici).
  3. Contieni gli eventi (debounce/throttle) e rimanda lavori non critici.

Domande frequenti

JS eccessivo o non differito aumenta il tempo di blocco e peggiora LCP/INP. Riduci, spezza e differisci il codice per migliorare le metriche.

Prima elimina il superfluo e applica code splitting; poi differisci o carica in modo condizionale gli script non critici.

Usa debounce per eventi che si ripetono rapidamente e vuoi eseguire solo al termine (es. ricerca digitata). Usa throttle per limitare la frequenza (es. scroll/drag) mantenendo reattività costante.

Gli script ES modules (type="module") sono defer di default e supportano import/export. Aggiungi un fallback nomodule per browser legacy: consente di servire bundle moderni più piccoli ai browser moderni.

Spezza i task lunghi, usa defer/async, carica on‑demand le feature, applica debounce/throttle e sposta lavori CPU‑bound in Web Workers.

Quando hai calcoli pesanti o parsing che bloccano il main thread (es. elaborazioni di dati, trasformazioni). I Worker spostano il carico off‑thread.

SSR e ISR riducono JS necessario al client e migliorano LCP; preferisci idratazione parziale/islands per limitare l’hydration globale.

Imposta soglie di peso (es. < 170 KB gz), TBT e INP. Automatizza in CI con Lighthouse CI/WebPageTest e interrompi la build quando superi i limiti.
